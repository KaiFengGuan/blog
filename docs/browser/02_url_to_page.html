浏览器输入URL到显示页面的过程

GKFTech 2022/4/29 浏览器浏览器原理

浏览器输入URL到显示页面的过程发生了什么？这是一道很常见的面试题，可以引申出非常多的知识点，常常作为考察基础前的必问题...... 在一次面试的时候我答得不是很好，现在整理一下，以免以后还出现问题。

大概来说，可以分为以下几个阶段：

  1. 输入
  2. 检查缓存
  3. URL解析
  4. DNS解析
  5. 三次握手建立TCP连接（HTTPS还要TLS握手）并发送请求
  6. 服务器处理并响应请求
  7. 四次挥手关闭TCP连接
  8. 解析响应数据
  9. 开始渲染

# 1 输入处理

  • 浏览器的UI线程会实时捕捉输入的内容，如果输入的不是网址或者协议不合法的话，就会使用浏览器默认的搜索引擎，来合成新的带搜索关键字的URL，准备进行搜索（这里如果有非法字符还会进行转义）
  • 执行一次当前页面的beforeunload事件，让页面退出之前执行一些数据清理工作，或者有表单没有提交的情况提示用户是否确认离开
  • 通过IPC把URL发送给网络进程

# 2 检查缓存

如果有缓存，并且没有过期，就不发送请求，直接拿来解码再开始渲染流程

  • 如果是https请求，有可能先找Service Worker，如果设置了请求拦截，离线缓存
  • 如果没有，再找浏览器的内存缓存（Memory Cache)
  • 如果还没有，再找硬盘缓存（Disk Cache）（ 强缓存和协商缓存都属于硬盘缓存）
  • 如果这三种都没有找到，请求还是http2的话，还可能会查找推送缓存（Push Cache)，就是找Session（Session会话结束就会释放，所以存在时间很短）
  • 如果没有缓存或者缓存过期，再开始解析URL

# 3 URL解析

  • 把请求需要的协议、域名、端口、路径这些信息解析提取出来
  • 根据解析出来的域名，进行DNS解析，找到要请求的服务器的IP地址

# 4 DNS解析

  • 本地DNS缓存查找（这个过程客户端只发出一次查询请求）
    • 先查浏览器缓存
    • 再查本地host文件
    • 然后查本地DNS服务器缓存
    • 如果都没有找到，就会走DNS服务器设置的转发器请求
  • 向跟根域名服务器发起解析请求（这个过程会发出多次查询请求）
    • 先去DNS根域名(.)服务器查询，属于哪个顶级域名服务器(.com)，然后返回顶级域名服务器IP
    • 再根据返回的IP去顶级域名服务器查找，属于哪个权威域名服务器(xxx.com)，返回权威域名服务器IP
    • 再根据返回的IP去权威域名服务器查找
      • 没有配置CDN，直接返回解析到的IP
      • 有配置CDN，权威域名会返回一个CName别名记录，它指向CDN网络中的智能DNS负载均衡系统，然后负载均衡系统通过智能算法，将最佳CDN节点的IP返回

# 5 建立TCP连接

三次握手的原因：

  • 阻止重复历史连接的初始化（主要原因）
  • 同步双方的初始序列号
  • 避免资源浪费（而不是四次握手）
  • 确认双方的收发能力（而不是二次握手）

三次握手过程：

  • 第一次握手：客户端向服务器发送（SYN，seq = x）
  • 第二次握手：服务器收到请求后向客户端发送（SYN + ACK，seq = y，ack = x + 1）
  • 第三次握手：客户端收到服务器的确认应答后，向服务端发送（ACK，seq = x + 1，ack = y + 1）

# 6 服务器处理请求

连接建立成功之后，浏览器会构建请求行、cookie等数据附加到请求头中，发给服务器，服务器接受请求并解析。

  • 如果没有对应资源，返回404状态码

  • 否则检查HTTP请求头有没有包含协商缓存信息（强缓存过期会走这个步骤），如果验证缓存没有更新，过期的缓存依然可以使用，就返回304状态码和空响应体

    • 要是没有缓存或者资源更新了，还没有CDN的话，就读取完整请求并准备http响应，进行查询数据库等操作
    • 要是连接的是CDN节点，并且正好有这个资源就直接返回，要是没有资源或者资源更新了的话，CDN服务器就去源站获取文件，如果源站也没有，就404了，有的话就返回给CDN节点缓存起来

  • 然后将响应数据通过之前建立的TCP连接，返回给浏览器的网络进程

  • 浏览器接收到响应数据之后，如果是http1.1以下则直接关闭连接，否则双方都可以根据情况选择关闭TCP连接或者保留重用，现在浏览器默认都会保持连接(keep-alive)

# 7 四次挥手关闭TCP连接

比如浏览器要关闭连接，过程是这样子的：

  • 第一次挥手：浏览器发送FIN报文，seq = u，并停止发送数据，但可以接收服务器的数据；
  • 第二次挥手：服务器收到后发送ACK报文，seq = v，ack = u + 1，表明收到；
  • 第三次挥手：服务器数据发送完了，给浏览器发送FIN + ACK报文，seq = w，ack = u + 1；
  • 第四次挥手：浏览器收到后，发送ACK报文，seq = w + 1，ack = u + 1。

# 8 解析响应数据

  • 如果返回的状态码是301或302就根据响应头的Location字段重定向到其他URL，然后一切从头开始，否则然后根据情况选择关闭TCP连接或者保留重用
  • 恶意站点检查：SafeBrowsing检查如果是恶意站点，就会展示警告页面
  • 如果状态码是2xx，然后判断资源是否缓存，是的话先缓存起来
  • 然后对响应解码（如gzip压缩），根据资源类型Content-type决定如何处理：
    • 如果是下载文件，请求提交给下载管理器；
    • 否则通知UI线程，创建渲染器进程来渲染页面。
  • 通过IPC管道将数据传给渲染器进程的主线程

# 9 渲染过程

渲染机制很复杂，需要执行的任务很多，渲染模块执行过程会被分为多个子阶段，开始一个边解析边渲染的流程：

# （1）解析HTML生成DOM树

  • 接收HTML，将原始字节数据转换为字符
  • 词法分析，将字符解析成标记，也称为令牌(token)
  • 语法分析，将标记转成DOM节点对象，并定义属性和规则
  • 解析器会维护一个解析栈，栈底为document，也就是DOM树的根节点，然后根据节点对象关系按顺序依次向解析栈添加，形成DOM树。

解析过程中遇到没有async或defer的script标签引用入时，会暂停解析过程，同时通过网络线程加载文件，文件加载后切换至js引擎执行相应代码，代码执行完成后再切换回渲染引擎继续渲染流程，也就是阻塞渲染。

# （2）解析CSS生成CSSOM树

  • 渲染引擎拿到CSS之后，首先格式化CSS，将CSS转为一个成浏览器认识的结构styleSheets；
  • 然后对计算好的样式进行标准化操作，比如rem、颜色(blue,red)、字体(bold)，转成统一的渲染引擎更容易理解的值；
  • 计算每个节点的具体样式信息，计算规则主要是继承和层叠（计算阶段的目的就是计算出DOM树中每个节点的位置信息、样式数据、文本节点数据）；
  • CSS解析和DOM解析是可以同时进行的
  • script执行和CSS解析不能同时进行，CSS会阻塞JS执行

因为JS执行时可能在文档的解析过程中 获取样式信息，如果样式信息没有加载和解析完毕，JS就会得到错误的值，所以会延迟JS执行

# （3）Layout布局

  • 首先创建布局树

    • 遍历DOM树中所有节点，将可见节点添加到布局树中，不可见的不包括，如meta、script、display:none...
    • 然后根据DOM结构和元素样式 对布局树中节点的几何位置信息计算

  • 分层（脱离文档流的元素会形成一个层叠上下文）

  • 然后主线程为每个图层计算样式，把每一个图层的绘制拆分成很多小的绘制指令，生成绘制表，这个表记录了绘制的顺序和绘制指令

# （4）栅格化

  • 主线程会把绘制表通过commit提交给合成器线程，合成器线程会再将图层分成图块；
  • 合成线程将分割好的图块发送给栅格化线程，然后分别栅格化每个图块，再将栅格化之后的图块存储在GPU内存中

合成器线程能够对不同的栅格化线程做优先处理，所以出现在视口内的图块会被优先栅格化

# （5）合成和显示

  • 当图块都被栅格化完成后，合成线程会收集栅格化线程的draw quads图块信息，该信息记录了图块在内存中的位置信息和图块在页面中的位置信息，这些信息会生成合成器帧
  • 合成器帧通过IPC传给浏览器进程的 viz 组件，这个组件用来接收这个合成器帧
  • 浏览器进程再将合成器帧绘制到显存中，再通过 GPU 渲染在屏幕上，这时候终于看到了页面内容
  • 当屏幕内容发生变化，比如滚动了页面，合成器线程就会将栅格化好的层合成一个新的合成器帧，新的帧再传到显存， GPU 再渲染到页面上
